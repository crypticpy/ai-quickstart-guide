Phase 3 is where the agency builds the infrastructure that lets developers ship AI features without becoming a security incident. The hard part is not standing up servers; it is doing so on whichever cloud, SaaS tenant, or approved managed platform the agency already uses, with enough safety, visibility, and repeatability for the risk tier. This phase is deliberately cloud-portable: the concepts apply across AWS, Azure, Google Cloud, and approved AI SaaS, but each implementation still has provider-specific identity, network, logging, and procurement details.

## Objective

Stand up a secure, isolated place where teams can experiment with AI workloads without risking production systems or exposing sensitive data. For small agencies this may start as an approved SaaS tenant or sandbox-only cloud setup. For standard and large agencies it grows into sandbox, staging, and production tiers where the Phase 5 platform can live.

## How this phase scales for your agency

<Tabs syncKey="agency-size">
<TabItem label="Small">

**Managed cloud services, SaaS-first.** Don't try to build a full sandbox/staging/prod stack from scratch. Use the approved managed offerings your agency already has, keep real data out of the sandbox, and skip the infrastructure-as-code investment until you have at least one engineer dedicated to platform work.

- **Use:** Approved AI SaaS or Azure AI Services / AWS Bedrock / Google Vertex AI directly behind your AUP. SSO through the identity provider you already have where available. One named admin, one secrets location, one cost view, and vendor or cloud logs.
- **Skip for now:** SLSA Level 2 attestations, OpenTelemetry pipeline, multi-environment promotion, custom landing zone. These are maturity adds, not prerequisites.
- **Budget:** $10K–$30K Year 1 is a planning range for managed-services consumption and minimal licensing; local pricing, contract vehicles, usage, and staff time can move it.
- **Off-ramp:** Sandbox-only is real; many small agencies pause here for 6–12 months while Phase 2 catches up. Document deferred items so they re-enter the roadmap when appropriate.
- **Where deferred items re-enter:** SLSA Level 2 attestations and the OpenTelemetry pipeline come back when Phase 5 starts (modules need build provenance and the AI orchestration layer needs AI-specific telemetry). Multi-environment promotion comes back when the Phase 6 starter project hits the production-readiness gate. The custom landing zone is a Year-2 hardening item, not a Year-1 prerequisite.

**Cloud sandbox following the full guide.** Build the sandbox + staging tier per the deliverables below, with one production environment ready when Phase 6 ships its first app. Use the per-cloud landing zone pattern, OpenTelemetry instrumentation, and SLSA Level 2 in CI/CD.

## Path guide

| Path | Do now | Use / simplify | Defer |
| --- | --- | --- | --- |
| Small | Approved SaaS or sandbox, SSO if available, one named admin, one secrets location, one cost view, vendor/cloud logs | Managed AI APIs and managed containers; existing admin console | SLSA, custom landing zone, full OTel pipeline |
| Standard | Sandbox, staging, CI/CD, secrets, observability, security baseline | Existing cloud landing zone and managed services | Advanced confidential computing |
| Large | Multi-environment substrate and production hardening | Dedicated platform/SRE team owns the substrate | Nothing core; defer only experimental hardening |

## Minimum viable infrastructure in plain English

For a small agency, Phase 3 can mean: staff sign in with the identity system they already use if SSO is available; one named owner administers the approved AI tool or sandbox; no real sensitive data is used in experiments; secrets live in one managed vault or approved admin console; logs and costs are visible; and prohibited data is documented plainly. The full standards stack comes later when the agency has more than one production AI workload.

## Procurement status changes quickly

AI coding assistants and model services change procurement status, data-residency options, and approved model lists quickly. GitHub Copilot's [FedRAMP model policy](https://docs.github.com/en/copilot/concepts/fedramp-models), [GitHub Enterprise Cloud data-residency documentation](https://docs.github.com/en/enterprise-cloud@latest/admin/data-residency/about-github-enterprise-cloud-with-data-residency), and April 2026 [FedRAMP/data residency announcement](https://github.blog/changelog/2026-04-13-copilot-data-residency-in-us-eu-and-fedramp-compliance-now-available/) are useful examples of how fast this area moves, not a universal procurement answer.

Before putting an AI coding assistant or model service into agency use, verify the current FedRAMP Marketplace or StateRAMP status where relevant, tenant type, data-residency region, policy settings, client requirements, data-use terms, and approved model list. State, local, tribal, and territorial agencies may also need SOC 2, CJIS-related addenda, HIPAA/BAA, grant terms, records requirements, or local procurement approval.

## Cloud-portable posture

Most agencies already have a primary cloud, typically the one with their existing identity tenant, data warehouse, or contact-center stack. The default of this guide is to **build on whichever cloud you are already on**, not to migrate. The architecture should treat the cloud as a substrate (compute, storage, identity, network) and isolate cloud-specific surfaces behind narrow adapters. The result is an agency that can pick the best AI service for the workload (AWS Bedrock, Azure OpenAI, Vertex AI, or a third-party API) without rebuilding the platform.

Three principles for staying cloud-portable:

1. **Pick managed services for commodities, code adapters for specifics.** Identity, secrets, container orchestration, logging, and metrics should usually be the provider's managed offering. Keep provider-specific code behind adapters so a future move changes the adapter and infrastructure, not the whole application.
2. **Use portable interfaces where practical.** OCI containers, OpenTelemetry, OpenAPI, structured logs, and standards-based OIDC/SAML federation reduce rewrite work. SPIFFE-style workload identity is a useful reference pattern for larger platforms, while many small and medium agencies will use the native managed-identity feature their cloud already provides.
3. **Be deliberate about serverless.** Lambda / Functions / Cloud Functions are excellent for thin glue and may be a reasonable first Tier-1 automation path. Keep the long-lived AI orchestration core (prompt management, retrieval, eval gates, rate-limit state) in a portable container when the workload is expected to grow.

Cloud-portable does not mean cloud-independent. Re-hosting still requires provider-specific IAM, network, logging, cost, procurement, and data-residency work. The goal is to make that work bounded and visible.

The guidance throughout Phase 3 is structured: pick the principle, then the per-cloud implementation. Where managed services diverge meaningfully (e.g., Azure Container Apps vs. AWS App Runner vs. Cloud Run), each cloud's section names the recommended option.

## Readiness gates

- **Before real data or production use:** Phase 1's baseline should be adopted: AUP, provisional risk tiering, approved-tools list, named owner, and clear data rules. Cloud provisioning and vendor terms should follow those decisions.
- **Before Tier-2 or Tier-3 pilots:** Risk classification must exist. Higher tiers need different network posture, logging retention, approval path, and contestation pathway than Tier-1.
- **Before broad developer access:** Confirm the human identity path. For some agencies this is a mature SSO provider; for others it is an interim admin-only setup while SSO is procured. Planning, local labs, and synthetic-data sandbox work can start earlier.

## What Phase 3 does NOT cover

- Application code or architecture (that's Phase 4–5).
- Specific model selection (Claude, GPT, Gemini, open weights). That decision happens in Phase 5 against the platform's AI orchestration adapter.
- Data warehouse and analytics infrastructure. These are upstream of AI workloads, but agencies typically already have them; integration patterns are referenced where relevant rather than rebuilt.
- Production deployment of starter projects. Phase 6 handles that against the infrastructure Phase 3 produces.

## Off-Ramp — Sandbox-only

Agencies that complete the sandbox tier (cloud account + SSO + minimal CI/CD + secrets + a single sandbox environment) have enough to run Track 4 developer labs and Tier-1 pilots. The full staging/production stack can be deferred to Year 2 if budget or staffing requires. This is a real off-ramp; many agencies will pause at sandbox-only for 6–12 months while culture work catches up. Document the deferred items so they re-enter the roadmap when appropriate.

## Plain-English Guide to Phase 3 Terms

- **Landing zone.** The pre-built scaffold of accounts, networks, and policies that any new workload lands into. Saves teams from configuring security from scratch each time.
- **Workload identity.** A way for software (not people) to prove who it is to other services without storing passwords. AWS calls it IAM roles; Azure calls it managed identity; Google calls it workload identity.
- **SBOM (Software Bill of Materials).** A list of every ingredient (library, dependency, version) in a piece of software, so you can answer "are we affected by the new vulnerability?" in minutes instead of weeks.
- **SLSA (Supply Chain Levels for Software Artifacts).** A graded supply-chain framework for build provenance and tamper resistance. Level 2 means a hosted build service generates provenance; Level 3 adds hardening. Phase 3 treats Level 2 as a standard/large-agency target and a small-agency maturity add.
- **Egress control.** Restricting outbound network traffic so a compromised workload cannot dial home to an attacker's server. Critical for AI workloads that often want to call external APIs.
- **Confidential computing.** Encryption that protects data not just at rest and in transit, but during processing. Available on all three major clouds; relevant for Tier-3 workloads.
