Phase 4 is where the agency decides _how_ it builds software. Phase 3 stood up the substrate; Phase 5 will produce the platform. In between, the agency picks a stack (or confirms the one it already has), writes down the standards that teams will follow, and establishes AI-assisted workflows that are already part of modern development. The goal is not to chase fashion. The goal is to make a small number of durable choices that the agency can defend in five years.

## Objective

Select (or validate) the agency's development stack and establish the coding standards, testing practices, AI-assisted workflows, and architectural records that will govern all platform and application development.

## How this phase scales for your agency

<Tabs syncKey="agency-size">
<TabItem label="Small">

**Pick one stack, adopt the starter baseline, get moving.** Small teams do not need a full standards program on day one. Confirm the stack they already operate, adopt the minimum repo template from this guide, write one lightweight ADR, and move on. The biggest gain in Phase 4 for small agencies is giving each developer or contractor a clear, safe path to use repeatable tooling.

- **Adopt now:** One primary stack, one repo template, formatter/linter/test command, secret scan, simple PR checklist, lightweight ADR template.
- **Configure carefully:** An approved AI coding assistant tenant, or a synthetic-code-only rule until procurement catches up.
- **Skip for now:** Deep contract-testing infrastructure, formal API registry, multi-stack debate, architecture council. One stack only.
- **Budget:** $5K–$15K is a planning range for tooling licenses and staff time; local procurement and existing contracts will move it.

</TabItem>
<TabItem label="Standard">

**Full deliverable list across Months 4–7.** Run the stack selection workshop, draft and adopt all standards, stand up the OpenAPI registry, and seed the ADR backlog. Sequence Track 4 (Developer Upskilling) to start mid-phase using the standards as they ratify.

- **Cadence:** Follow the 12-week sequencing schedule below.
- **Reviewer training:** Pair Track 4 cohorts with the AI-assisted dev policy rollout. Reviewers need explicit calibration.
- **Budget:** $15K–$40K is a planning range for tooling licenses, facilitator time, and cohort training.

</TabItem>
<TabItem label="Large">

**Full deliverable list plus inner-source contribution standards and stack-secondary support.** Add a formal RFC process for stack additions, a cross-team architecture council that reviews ADRs, and inner-source contribution guidelines for module changes. Run two parallel reference implementations if the agency genuinely supports two stacks.

- **Add:** Architecture council (monthly), inner-source contribution guide, cross-team ADR review, dedicated developer experience (DevEx) function.
- **Cadence:** All Standard items, plus DevEx team stands up parallel to the standards work.
- **Budget:** Often skews past $40K once dedicated DevEx staffing, cross-team governance, and tooling are included.

</TabItem>

## Path guide

| Path | Do now | Use / simplify | Defer |
| --- | --- | --- | --- |
| Small | Pick one stack, adopt starter standards, configure AI coding tools safely or restrict them to synthetic code | One ADR that ratifies defaults; one repo template | Formal API registry, Pact Broker, multi-stack review |
| Standard | Run stack selection, standards, testing, API, ADR, and AI-dev workflows | Use the reference implementation as the standard example | Secondary-stack support |
| Large | Add architecture council, RFCs, and DevEx ownership | Support a second stack only if already real | New stack experiments without a business owner |

## Why this phase, why now

Most agencies under-invest in this phase. The pull is to skip from "we have infrastructure" to "build the thing." Skipping produces predictable pathology: each team picks a different stack, each PR review is litigated from scratch, AI tooling is configured ad hoc with policy drift between teams, and the platform ends up unmaintainable because nothing is shared. Phase 4 is two months of writing things down to save twelve months of cleanup.

The phase is also the first place AI-assisted development gets serious agency-wide attention. The [2025 JetBrains State of Developer Ecosystem](https://blog.jetbrains.com/research/2025/10/state-of-developer-ecosystem-2025/) reports 85% of developers regularly use AI tools and 62% rely on at least one AI coding assistant. With Claude Code, GitHub Copilot, and Cursor in mainstream developer use, the question is no longer "do we allow this?" but "how do we configure it safely, and what do we expect from a reviewer?"

The 2026 [JetBrains AI Pulse](https://blog.jetbrains.com/research/2026/04/which-ai-coding-tools-do-developers-actually-use-at-work/) makes the trend clearer: a January 2026 wave of more than 10,000 professional developers found 90% regularly used at least one AI tool at work for coding and development tasks. Treat that as a market signal, not a guaranteed agency outcome. Measure local value through PR cycle time, defect rate, reviewer load, developer feedback, and whether more work ships without lowering quality.

## Minimum useful Phase 4

If the agency has no dedicated software platform team, Phase 4 can still produce a useful stopping point:

| Area | Minimum useful version |
| --- | --- |
| Stack | Confirm the stack already used by staff, vendors, or the hosting platform; avoid a migration unless there is a concrete blocker |
| Repo template | One starter repo with `README`, local run command, test command, formatter/linter, secret scan, and `.env.example` |
| Review | One-page PR checklist: what changed, how tested, data handled, risk, reviewer |
| ADR | One lightweight ADR that records the chosen stack and why |
| AI coding tools | Approved tenant and settings, or synthetic/sample-code-only use until approval exists |
| Data | Synthetic data only in local/reference work |
| API | Committed OpenAPI file or README contract table for any API the pilot exposes |

## Readiness gates

- **Planning and local labs can start early.** Stack selection, coding standards, ADRs, and a local reference implementation can begin while Phase 3 is still maturing, as long as teams use synthetic data and local/dev-only credentials.
- **Before broad developer access or real data:** Phase 3 identity, secrets, logging, and approved tool access should be in place.
- **Before production deployment:** CI/CD, deploy controls, eval/test gates, and the relevant Phase 3 environment need to be operational.
- **Track 4 (Developer Upskilling) cohorts.** Phase 4 outputs feed Track 4 directly. The labs use the agency's reference repo, the agency's coding standards, and the agency's AI-assisted dev configuration where available. Sequence Track 4 to start mid-Phase-4.

## What Phase 4 does NOT cover

- The AI orchestration layer's internal architecture (that's Phase 5).
- Specific module designs (Auth, Data Grid, etc.); Phase 5 again.
- Operational runbooks. Those emerge during Phase 5–6 from the platform's actual operations.
- Choosing a foundation model. Happens against the platform's adapter, not the dev stack.

## Off-Ramp — Stack-Conformant Pilots

Agencies that complete the Phase 4 baseline, even without Phase 5's full platform, have enough to ship Tier-1 pilots with a repeatable quality posture: coding standards, basic tests, approved AI-assisted development rules, and ADR discipline. A two-person team can often build a useful internal AI feature against the agency's reference implementation in 2–4 weeks. Production use still depends on the relevant Phase 3 gates and governance approval. Many agencies should pause here for a quarter, ship 2–3 pilots, and let the lessons inform Phase 5 module design.

## Plain-English Guide to Phase 4 Terms

- **Stack.** The set of programming languages, frameworks, and runtimes the agency commits to maintaining. Picking one (or two) means every team can read each other's code and share modules.
- **Linting and formatting.** Automated tools that check code for common bugs (linting) and reformat it consistently (formatting). The bar should be "the tool runs on save and in CI; nobody hand-formats."
- **Contract testing.** Tests that verify two services agree on the shape of the messages they pass, without running both services in the same test. Catches integration breakage before deploy.
- **OpenAPI specification.** A machine-readable description of an API's endpoints, request shapes, and response shapes. Becomes the source of truth that client SDKs, server stubs, documentation, and tests all generate from.
- **ADR (Architecture Decision Record).** A short document that captures _what_ was decided, _why_, _when_, and what alternatives were rejected. Future engineers find ADRs when they ask "why did we do this?" Without them, the answer is folklore.
- **AI-assisted development.** A developer working with an AI tool (Claude Code, Copilot, Cursor) that reads the codebase and proposes changes. The output is code, not prose; the discipline is the same code review.

API-first design adoption research, modular monolith case studies, AI-assisted development surveys, Trunk-Based Development from State of DevOps reports, the SPACE framework for developer experience, and the ADR pattern as documented by Michael Nygard. These sources are reference points; agencies should validate local impact with their own delivery, quality, and support metrics.
