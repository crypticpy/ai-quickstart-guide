# AI Deck Source Inventory

Use this inventory with `AI_DECK_SOURCE_GUIDE.md` when creating AI-importable presentation markdown for the guide.

The purpose is not to turn every page into slides. We should create deck sources where the guide already calls for a presentation, briefing, webinar, launch meeting, sponsor report, or public/board-facing discussion. Hands-on labs and working sessions should stay activity-led unless a short framing deck would help smaller agencies run the session.

## Lower Priority Or Do Not Create By Default

These areas may mention demos, dashboards, videos, or presentations, but a deck source is not the right default artifact.

| Area | Recommendation | Reason |
| --- | --- | --- |
| Track 4 Developer Upskilling labs | Do not create full decks by default. Consider short instructor intros only after lab guides stabilize. | The track explicitly produces shipped code, not slides. |
| Phase 3 Infrastructure pages | Do not create standalone decks by default. | These are implementation guides; diagrams and checklists matter more than decks. |
| Phase 4 Dev Stack pages | Do not create standalone decks by default. | ADRs, standards, and reference implementation docs are better artifacts. |
| Phase 5 Platform module pages | Do not create standalone decks by default. | Platform pages are technical module specs; deck summaries could be useful later for architecture councils, but not first. |
| Case study pages | Do not create separate decks initially. | Reuse case study content inside Track 1, Track 2, budget, and oversight decks. |
| Gantt and Dependencies page | Do not create a separate deck initially. | The roadmap should be pulled into kickoff, Track 2.4, budget, quarterly report, and selection decks. |
